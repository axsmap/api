AXS Map pledge settlement and Salesforce synchronization
=========================================================

Salesforce object
-----------------

AXS_Map_Pledge__c

Required additional field
-------------------------

Create these fields before enabling synchronization.

Pledge:
Label: AXS Map Pledge ID
API name: AXS_Map_Pledge_ID__c
Type: Text
Length: 24 or greater
Options: Unique and External ID

Campaign:
Label: AXS Map Mapathon ID
API name: AXS_Map_Mapathon_ID__c
Type: Text
Length: 24 or greater
Options: Unique and External ID

Opportunity:
Label: AXS Map Donation ID
API name: AXS_Map_Donation_ID__c
Type: Text
Length: 24 or greater
Options: Unique and External ID

These fields provide duplicate-safe Pledge upserts and exact Campaign
matching. Campaign Name is never used as a fallback.

Every Salesforce Campaign representing a Mapathon must contain the matching
MongoDB event ID in AXS_Map_Mapathon_ID__c before its pledges are synchronized.

Contact permission policy
-------------------------

The integration treats Salesforce Contacts as read-only:

- Donors and Mapathon participants are looked up by email.
- The integration never creates, edits, or deletes Contact records.
- If a donor Contact is not found, Donor__c is left blank and
  Donor_email__c is still populated.
- A participant Contact is required because Mapathon_Participant__c is part
  of the pledge relationship model. A missing participant Contact causes the
  pledge sync to fail for administrator review.
- A first-time donor is still stored in MongoDB and can be synchronized to
  an Opportunity without creating a Contact. The donor email is written to
  the configured Opportunity donor-email field.

AWS secrets
-----------

Store Salesforce credentials in the staging API environment or AWS Secrets
Manager. Do not commit them to Git.

API workflow
------------

GET /donations/pledges/settlement-candidates
POST /donations/pledges/:donationId/close
POST /donations/pledges/:donationId/sync-salesforce
POST /donations/:donationId/sync-salesforce
POST /donations/salesforce/payment-confirmation

The settlement, close, and manual synchronization routes require an
authenticated AXS Map administrator. The payment-confirmation route is
server-to-server and requires these headers:

x-axsmap-timestamp: current Unix epoch time in milliseconds
x-axsmap-signature: lowercase hex HMAC-SHA256 signature

The signed value is:

timestamp + "." + axsMapPledgeId + "|" + status + "|" + opportunityId +
"|" + paypalTransactionId + "|" + paymentDate + "|" + receiptSent

The HMAC key is SALESFORCE_WEBHOOK_SECRET. Requests older than five minutes
are rejected. Paid updates require an Opportunity ID, PayPal transaction ID,
and payment date. MongoDB uses the frozen pledge calculation and does not
trust Salesforce to supply a different amount.

Flat donation workflow
----------------------

After PayPal confirms a flat donation, AXS Map keeps the MongoDB payment
confirmation even if Salesforce is temporarily unavailable. It then attempts
to synchronize an Opportunity. The synchronization first searches the
configured PayPal transaction field and updates that Opportunity if found;
otherwise it upserts by AXS_Map_Donation_ID__c. This prevents duplicate
Opportunities when another PayPal automation created the record first.

The configured PayPal transaction field should be marked Unique in
Salesforce where possible to protect against concurrent duplicate creation.

Salesforce automation still required
------------------------------------

There are currently no Flow Triggers on AXS_Map_Pledge__c and no pledge flow
was found in the Salesforce Flow inventory. Salesforce still needs an active
automation that sends the payment-request email and updates Status__c from
Calculated to Payment Requested.

Salesforce also needs an authenticated Flow/Apex callout that posts pledge
payment and receipt updates to:

POST https://<stage-api-host>/donations/salesforce/payment-confirmation

The callout must generate the timestamp and HMAC headers described above.
